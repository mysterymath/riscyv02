Now, we'd *immediately* like to begin fetching the next instruction. That means
that we lose access to the opcode. That means that you can't just place the
opcode somewhere; it's a PIPELINE.

What that means is that each stage will have a set of registers as its inputs
and a set of registers as its outputs. So, what are these for the fetch stages?

fetch lo: inputs: none, outputs: inst lo
fetch hi: inputs: inst lo, outputs: inst lo, inst hi

Now, if we actually made registers for these, we could, without other concerns,
start fetch lo of the next insructoin immediately after fetch lo of the
previous:

fetch lo
fetch hi | fetch lo

That would work fine on the inside, but we can't actually do two fetches at the
same time on the outside. In such cases, we must defer issuing the next
instruction; and that means we're free to **pass through** an input to the
output using only a single register.

So, let's just name the next stage next, and decide it doesn't do anything.

fetch lo: inputs: none, outputs: inst lo
fetch hi: inputs: none, outputs: inst lo, inst hi
next:   inputs: inst lo, inst hi, outputs: opcode, rd1, rs1, rs2, imm | fetch lo: inputs none, outputs: inst lo

Now, it's fine to have an output and input of the same reg at the same time;
one completes as the clock; the other begins at the clock, so they don't
actually overlap.

Our do nothing next stage needs to output all of the parts of the instruction,
since it doesn't do anything. We're not aiming for a very high fmax, because
our old slow DRAMs are old and slow. The goal is to produce a better 6502, so
we want as much work in each pipeline stage as we can stomach. Only needs to go
at 1MhZ, after all!

The more next does, the fewer outputs it needs. Our goal is to do an ALU
operation, but to do that, we may need up to 2 register operands. So, it should
fetch the first one.

Here's a fun way to think of a conditional branch. At the end of the execute

Okay, so jumps, loads/stores, and branches need to manipulate the pipeline. The
need arises in execute for branches and decode for the others.

For a jump, the fetches during its decode and execute would be wrong. The
correct next instruction isn't known until execute completes. Accordingly, the
decode step should cause the next fetch (during execute) to see a NOP as input
instead of fetching, and it should cause the concurrent fetch step to output a
NOP instead of what it fetched.

For a branch not taken, it has no effect; it's like an arithmetic instruction.

For a taken branch, we could logically think of its execute step as inserting a
new JAL x0, tgt into the pipeline to be fetched, decoded, and executed.
However, this determination is made when the execute of the branch finishes. At
that point, the fetch and decode for that logical instruction have just
finished, based on the instruction after the branch. That's perfect; the
execute of the branch is concurrent with the decode of the logical JAL. We can
have the taken branch cause the effects of decoding a JAL, except that it's
triggered by the branch condition, not the opcode. That's all the branch
execute step needs to do to effect the branch.

Similarly, we can have the decode phase of a non-zero shift change the output
of the fetch stage to be that of a shift with a lesser amount. Each stage would
execute in sequence, so it would take between 2 and 14 cycles to issue a shift.
We could also use this trick to convert variable shifts into fixed ones; this
has no penalty, since the first shift operation can be done in the first

What would this look like if it had a two stage pipeline instead? That's the
minimum needed to support two cycle execution. We trade latency and another
register file read port for reduction in state passed between stages. Seems
wroth(tm).

The stages would be, of course, fetch and execute. Fetch would know the opcode
at the start of its first cycle. We'd want to have a predicted branch target
ready by the next cycle. However, we don't know the branch offset until the
start of the second cycle. That means we need an 8-bit adder with 4-bit CLA
(a real one) and a 8-bit incrementer.

Okay, so if a branch is predicted, it's actually taken by the time the next
fetch starts. We save the non-predicted branch pass and hand it to execute
along with the instruction.

Execute first decodes the instruciton, then sets up the ALU for the first byte.
We store the 8-bit ALU output; we can't write it back, since the ALU may be set
up in a loop with the register file.

On the second cycle, the ALU operation is complete. If it indicates that the
branch was mispredicted, then the concurrent fetch cycle is made to register a
NOP and the PC is set to the non-predicted branch. If both fetch and execute
try to branch, that means that there definitely was a mispredicted branch,
which means the fetch is wrong. So branch PC wins over fetch.

If the operation is a load or store, then execute passes control to the
load/store unit. This keeps fetch and execute from processing their inputs
until the load or store has finished.

This gives a 1 or 2 cycle penalty for loads/stores, no (!!!) penalty for
predicted branches, and a 2 cycle penalty for mis-predicted branches.

(they could be thought to latch on posedge or negedge of w, respectively). It
technically just needs to be after w falls, but we want to mirror the 6502's
footprint, so we need to respect the ordering of its timing diagram.

Okay, let's steal one bit from LB and friends to indicate sp vs ptr. Ideally
we'd take a bit from the immediate, but that would mean adding a bit to LUI
and AUIPC somehow (not JAL).

Without upsetting our current opcodes, that would reserve all opcodes ending in
1100 and 1101. That kills AND and OR. But we have two bits free at the top of
those opcodes; no worries.

Let's start looking at our encoding from most fixed to least fixed.

How much does a reg reg instruction cost? Well, each uses 9 bits out of 16 for
register operands.  There are 10 of them, so they take 5120 instructions out of
65536. That leaves 60416.

Our loads and stores take up 2^11 instructions, and there are 5, for 10240 total.
That leaves 50176.
 
Our imm operations take 1024 or 2048 instructions each for 7 or 8 bit
immediates, and there are 14. That totals to 14336 or 28672, leaving 35840 or
21504.

Our upper immediate operations take 4096 or 2048 instructions each, and there
ar three. That leaves 23552 or 15360.

Let's do this symbolically. The reg reg bits are fixed, so discount them. We
have 60416 instructions.

Loads and stores each take 2^(3 + ldrhs + immlo). Immediate operations each take 2^(3 + immlo).
Upper immediate operations take 2^(3 + (16 - immlo)) = 2^(19 - immlo).
So, we must have that: 14*2^(3+immlo) + 3*2^(19 - immlo) + 5*2^(3 + ldrhs + immlo) <= 60416.

Now ideally, ldrhs = 3. Plugging that into wolfram alpha, it claim that immlo=6
is a solution. That would be nice: we could access structs of up to 32 bytes in
a single instruction. But branches would be rough.

Let's break out branches. We then have:

12*2^(3 + immlo) + 2*2^(3 + brimmlo) + ...

Now ideally, we would have ldrhs = 3. What are the implcations of the other
values? We could have a max immlo of 6, but a brimmlo of 9. That sets the stack
and struct size to 32 bytes, and the branch range to +-256 instructions. Not bad.

How about a ldrhs = 2? Then we could have brimmlo of 10, which is almost too
much: 512 instructions. We can have an immlo of 7 at a brimmlo of 9. Both are quite execellent:
struct range of 64 bytes, and a branch range of 256 instructions.

A naive ldrhs of 2 would allow us to access x0 (zero), x1(link reg), x2(stack
pointer), and x3. x1 is often free, so it and x3 are general pointers. x0 gives
access to a zero page and high page, and x2 the stack. Seems like a good deal,
honestly, and it should be loose enough for a register allocator to deal with.

ldrhs = 1 buys a ldimm of up to 8; that's what we expected. But then we can
only have sp and a pointer; we lose the sexy sexy zero and I/O pages.

ldrhs = 0 doesn't expand brimm or immlo; no need to bother.

Hmm, an immlo of 6 exactly matches the proportion of RISC-V's 12-bit immlo to
32-bit addresses. That's appealing.

So, let's queue it up for ldrhs=2, brimmlo=9, immlo=7. Our instruction widths become:
imm: 10, br: 12, ui: 12, ldst: 12, reg reg: 9.

How much weight does x0 pull?
- You can use it with loads and stores to access the zero/top of mem pages
- It's not useful with the immediates at all
- A SUB from zero negates.
- A SLT x, x0 produces an is-negative result... but so does SLTI x, 0
- A SLT x0, x produces an is-positive result... but so does !SLTI x, 1
- Storing 0 to memory; that's quite useful.

The above is kind of middling; you can always just set a register to 0 with LI
0. And we only have 8 of them; it really does seem like that doesn't pull
enough weight to make it worthwhile.

Of the above, it only maybe seems worth adding a NEG opcode, but it would be
the only unary opcode present; so it would ignore lll. Seems like a wart, and
LI x0, 0; SUB x0, 0, x0. You can LI x0,0 at the top of a function if you want.

Okay, let's collect a list of all of the things that have bearing on interrupts
in RISC-V U and M specs.

The CSRs seem important. The following operations are atomic

- CSRRW: atomically swap rd (original has rs1; no need) with CSR
- CSRRS: write current value to rd, OR rs1
- CSRRC: write current value to rd, AND ~rs1
- The immediate versions of the above, which zero extend rs1 and treat it as an immediate.

What CSRs are relevant?

- mstatus
  - ie: Interrupt enable. Doesn't effect NMI. Set to 0 on trap.
  - pie: Value of interrupt-enable before trap. We do actually need this, since
    the trap could be a NMI, so this could be either 0 or 1.
- ip
  - One bit; NMIs are handled separately. We don't actually need this; the 6502 doesn't have it.
- ie:
  - One bit: NMIs are handled separately. Actually, we don't need this at all; it's completely redundant with ie.
  - We could allow masking software interrupts, but if you can't do it on a
    65c02, it's not in scope unless we want it to be.
- epc:
  - Program counter for interrupt return. We need this. Low bit is always zero,
    but not shifted. Instructions should be atomic WRT interrupts; this should
    always point the the instruction that hasn't yet been executed. This may
    need tricky handling for side effects in the fetch stage and multi-cycle
    execute stages.
- mcause:
  - Whether it's a software interrupt, IRQ, or NMI.

- No way to indicate NMI on 6502, but separate trap vector. Let's just do that too. 

- We can store ie, pie, and brk in one 3-bit register. We need a 15-bit
  register for epc. Let's have a 1-bit CSR selector between the two. We don't need
  to allow masking bits on epc, only atomic exchange.

- We'll take a cue from RISC-V and state that NMI handlers are intended *only*
  for system error conditions, e.g. warm restart. NOT for fast interrupts. Thus, they
  set epc and cause, but they don't affect pie or ip. 

- We can require that interrupt premption be done in software. Low priority
  interrupts should clear their pending state and immediately reenable
  interrupts. High priority interrupts should clear their pending state and
  execute with interrupts disabled.

- Let's consider the antic DLI. The 6502 takes 8 to 14 cycles to respond to the
  interrupt. We would take 2 to 3 cycles. The handler routine takes 11 machine
  cycles. We would need, at minimum, a load of the status, a branch on that
  status, and an absolute call. Thats 9 cycles! We guuud.

- But, saving/restoring 16-bit registers on interrupt could be a problem... it
  slow.
- But is it? PHA TXA PHA TYA PHA is 13 cycles. ADDI x1, -6, SW x1, x2, 0, SW
  x1,x2,2, SW x1,x2,4 is... 14 cycles. Wow!

Relevant instrucitons:

- iret:
  - Sets ie to pie. Sets pie to 1.
- ebreak:
  - Software interrupt.
